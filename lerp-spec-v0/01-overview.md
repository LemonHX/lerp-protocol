# 01 · Overview

## 设计哲学

### 问题：现代网络的寻址困境

传统网络连接需要知道：IP 地址、端口号、防火墙是否放行、对方是否在 NAT 后面、NAT 类型……这些都是"网络拓扑"的概念。网络拓扑随时变化，导致连接随时失效，且对用户完全不透明。

### lerp 的回答：Ticket 即一切

```
传统网络：我要连 192.168.1.1:8080
lerp：    我有这张 ticket
```

ticket 是一张**能力凭证**——持有它就能连，不持有就连不上。对方换 IP、换运营商、从家到公司，ticket 不失效。端口是多少、NAT 有几层，用户无需关心。

| 传统网络 | lerp |
|---|---|
| 需要知道对方 IP + 端口 | 只需要 ticket |
| 防火墙/NAT 需要手动穿透 | 自动处理 |
| 对方换 IP 则连接失效 | ticket 不失效 |
| 端口冲突需要协调 | 与端口无关 |

---

## 职责边界

lerp 只做一件事：**在两个 endpoint 之间建立端到端加密的双向流。**

管道里跑什么，lerp 完全不关心：

- 数据同步协议（CRDT、gossip、etcd……）→ 应用自决
- 服务发现、成员管理 → 应用自决
- 消息格式、序列化 → 应用自决

**lerp 不是框架，是管道。**

---

## 术语定义

| 术语 | 定义 |
|---|---|
| `endpoint` | lerp 网络中的一个节点，由一个 Ed25519 密钥对唯一标识 |
| `endpoint_id` | endpoint 的 Ed25519 公钥，32 字节，全局唯一，无需注册 |
| `relay` | 中继服务器，负责在两个 endpoint 之间盲转发加密流量 |
| `relay_secret` | endpoint 与某个 relay 共享的对称密钥，用于派生路由令牌 |
| `routing_token` | 由 `endpoint_id` 和 `relay_secret` 派生的短期路由标识，放入 SNI |
| `ticket` | 包含连接所需全部信息的能力凭证，out-of-band 分发 |
| `lerp-daemon` | lerp 的系统守护进程实现，运行于 Linux/k8s 等环境 |
| `lerp-client` | lerp 的浏览器实现，暴露 Web Streams API |
| `time_bucket` | 时间窗口编号，`floor(unix_timestamp / window_seconds)`，用于令牌轮换 |

## 协议栈

```
+----------------------------+
|       应用层协议            |  ← 用户自定义，lerp 不关心
+----------------------------+
|   lerp E2E 加密双向流       |  ← lerp 的实际产物
+----------------------------+
|   WebTransport / QUIC      |  ← 传输层
+----------------------------+
|   TLS 1.3 (SNI 路由令牌)   |  ← 用于 Relay 路由 + 传输加密
+----------------------------+
|      UDP (443 port)        |  ← 网络层（HTTP/3 over QUIC）
+----------------------------+
```
